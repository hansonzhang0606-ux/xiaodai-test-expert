# 效贷功能测试专家 — 测试人员使用指导手册

> **版本**：v1.3.4 | **适用对象**：效贷业务线功能测试人员 | **更新日期**：2026-07-22

---

## 一、专家简介

**效贷功能测试专家**是效贷业务线专用的 AI 测试助手，内置从需求文档到测试用例入库的端到端工作流，帮助测试人员高效完成测试设计工作。

### 能做什么

| 能力 | 说明 |
|------|------|
| 需求文档整理 | 将多格式需求文档（Word/PDF/图片/Excel）或 Confluence 在线页面统一整理为结构化 Markdown |
| 需求评审 | 6 维度自动评审（完整性/一致性/边界/异常/优先级/可测性），输出评审报告 |
| 测试点生成 | 基于需求+评审报告+知识库，推导测试点并输出 XMind 脑图 |
| 测试用例生成 | 将评审后的测试点细化为可执行的测试用例 Excel |
| 知识入库 | 将本次需求的测试经验沉淀到知识库，供后续复用 |
| 时间节省追踪 | 自动记录每一步为测试人员节省了多少时间，生成团队效能统计报告 |

### 谁能使用

效贷测试专家通过**花名册**控制访问权限。管理员维护一份在职人员名单，只有名单内的人员才能使用。

**专家不会展示人员列表**——你需要输入自己的姓名进行身份验证，匹配通过才能使用。如果你输入姓名后提示"不在花名册中"，说明你尚未被授权，请联系管理员添加。

> 这一设计是为了保护团队信息安全，也为后续发布到专家市场做准备。

## 二、快速开始

### 2.0 如何安装专家（首次使用）

效贷测试专家已发布到团队 GitHub 仓库，首次使用需要安装：

**第 1 步：添加团队市场**

在 WorkBuddy 中，进入**设置 → 市场管理 → 团队市场**，添加仓库地址：

```
https://github.com/hansonzhang0606-ux/xiaodai-test-expert
```

添加后，WorkBuddy 会自动拉取仓库中的专家列表。

**第 2 步：安装专家**

进入**专家中心**，找到 **"效贷测试专家"**，点击**安装**。安装完成后即可在对话中使用。

> 安装过程是全自动的，不需要额外配置。如果你在专家中心看不到"效贷测试专家"，请确认：
> 1. 团队市场已正确添加（检查仓库地址是否拼写正确）
> 2. 管理员已在 GitHub 仓库中邀请你为 Collaborator
> 3. 网络可以正常访问 GitHub

**第 3 步：生成个人 PAT（用于工时数据同步）**

为了让你的时间节省数据能同步到团队数据仓库，你需要一个 GitHub Personal Access Token（PAT）：

1. 登录 GitHub → Settings → Developer settings → Fine-grained tokens → Generate new token
2. **Token name**：`xiaodai-expert-data`
3. **Repository access**：选择 `Only select repositories` → 选择 `hansonzhang0606-ux/xiaodai-test-expert`
4. **Repository permissions**：
   - Contents → **Read and Write**
   - Metadata → **Read**（自动勾选）
5. 生成后**立即复制** token，保存到安全位置（如密码管理器）
6. 将 token 提供给效贷测试专家（在对话中告知），专家会配置自动同步

> ⚠️ PAT 是你的个人凭据，**不要分享给他人**。如果泄露，请立即到 GitHub 设置中撤销并重新生成。

### 2.1 如何打开专家

在 WorkBuddy 中，进入专家中心，找到 **"效贷测试专家"**，点击进入对话。

### 2.2 首次对话：验证身份

每次新开对话，专家会**首先要求你输入姓名进行身份验证**：

```
👋 欢迎使用效贷测试专家。请输入你的姓名？
```

**你只需要输入自己的真实姓名即可**，例如回复 `吴香康`。

| 情况 | 结果 |
|------|------|
| ✅ 姓名在花名册中 | 验证通过，专家欢迎你并开始服务 |
| ❌ 姓名不在花名册中 | 提示"不在效贷测试团队花名册中，无法使用本专家"，需联系管理员添加 |

**安全说明**：
- 专家不会展示人员列表，避免暴露团队信息
- 姓名需精确匹配（不要输入昵称、简称）
- 只有管理员在花名册中添加的在职人员才能使用
- 身份验证后，整个会话期间不需要再重复输入姓名

### 2.3 四个快捷入口

专家提供了 4 个快捷按钮（Quick Prompts），点击即可快速发起：

| 快捷按钮 | 作用 |
|---------|------|
| **帮我整理这个需求目录下的文档** | 从步骤①开始，整理本地需求文档 |
| **提取这个 Confluence 页面内容并整理成需求文档** | 从步骤①开始，提取 Confluence 页面 |
| **评审这份需求，指出风险和缺失点** | 从步骤②开始，评审已整理的需求 |
| **查看时间节省统计** | 查看全团队的时间节省分析报告 |

当然，你也可以直接用自然语言描述你的需求。

## 三、两种使用模式

### 模式 A：完整流程（新需求从零开始）

适用于：拿到一个新需求，需要从头到尾走完整流程。

```
① 文档整理 → ② 需求评审 → ③ 确认评审 → ④ 生成测试点 → ⑤ 评审XMind → ⑥ 生成用例 → [⑦ 入库知识库]
```

**怎么触发**：
- 提交需求目录，说"走完整流程"或"从需求到归档"
- 发送 Confluence URL，说"提取这个页面内容，走完整流程"

**执行方式**：串联执行，每完成一个阶段会通报进度并等待你确认，不会自动跳到下一步。

- 本地目录 → 专家会扫描目录、转换文件、生成整理版 MD
- Confluence URL → 专家会调用 confluence-to-md 能力提取页面、生成整理版 MD

### 模式 B：单步模式（按需执行某一步）

适用于：已经有部分产出物，只需要执行某个特定步骤。

**常见场景**：

| 场景 | 你说的话 |
|------|---------|
| 只需要整理文档 | "帮我整理这个目录下的需求文档" |
| 只需要评审需求 | "评审这份需求" |
| 已有评审后 XMind，要生成用例 | "我有评审后的 XMind，帮我生成用例" |
| 把测试经验归档到知识库 | "入库" |

## 四、7 步工作流详解

### 步骤 ①：文档整理

**你说**："帮我整理这个需求目录下的文档" / "处理这些文档"

**你提供**：包含需求文档的目录路径（支持 Word、PDF、图片、Excel 等多种格式）

**专家产出**：
- 整理版 Markdown 文件（结构化、带锚点溯源）
- 原始文件归档目录

**你需要做的**：检查整理版 MD 是否准确，确认后进入下一步。

---


### 步骤 ①（替代入口）：Confluence 页面提取

**你说**："帮我提取这个文档内容" / "整理这个 Confluence 页面"

**你提供**：Confluence 页面链接，例如：

```
https://your-confluence-server/pages/viewpage.action?pageId=123456
```

**专家产出**：
- 整理版 Markdown 文件（结构化、带锚点溯源）
- 页面中的图片/附件归档到 `source/images/`
- 整理索引 `source/.整理索引.yaml`

**你需要做的**：
1. 检查整理版 MD 中的正文内容是否完整
2. 核对页面中的图片/附件是否正确归档
3. 确认无误后回复「确认整理完成」
4. 说「评审这个需求」进入步骤②

> **与本地目录整理的关系**：两个入口**二选一**。Confluence 提取完成后生成的整理版 MD 与本地整理的完全等价，都可以直接进入步骤②。

---

### 步骤 ②：需求评审

**你说**："评审这份需求" / "评审这个需求"

**专家产出**：
- 评审报告 Markdown（6 维度分析：完整性/一致性/边界/异常/优先级/可测性）
- 评审数据 JSON

**你需要做的**：阅读评审报告，确认评审结论是否合理。

---

### 步骤 ③：确认评审

这是一个人工确认环节。你确认评审报告没有问题后，告诉专家"确认"或"没问题"，才会进入下一步。

---

### 步骤 ④：生成测试点

**你说**："生成测试点" / "转 XMind"

**专家产出**：
- 测试点 XMind 脑图文件
- 测试点 JSON 数据文件
- 测试点生成报告

**你需要做的**：打开 XMind 文件，评审测试点是否完整、准确。

---

### 步骤 ⑤：评审 XMind

这是一个人工评审环节。你在 XMind 软件中直接修改测试点脑图（增删改节点、标记疑问等），修改保存后告诉专家"XMind 评审完了"。

---

### 步骤 ⑥：生成用例

**你说**："生成用例" / "生成 Excel"

**专家产出**：
- 测试用例 JSON 文件
- 测试用例 Excel 文件（可直接用于测试执行）

**你需要做的**：检查 Excel 中的用例是否准确、可执行。

---

### 步骤 ⑦：入库知识库（可选）

**你说**："入库" / "归档"

**专家产出**：
- 知识库更新（历史需求、评审报告、技术方案、测试用例归档）
- 精华库提炼（高频根因/遗漏/改进措施 Top 10 更新）
- 进化日志更新

> 这一步是可选的。如果你觉得这次需求有值得沉淀的经验，说"入库"即可。

## 五、时间节省追踪（重要）

### 5.1 为什么需要你反馈时间

专家的核心价值之一是**为测试人员节省时间**。为了量化这个价值，每完成一个关键步骤后，专家会**强制询问**你这一步节省了多少时间。

这些数据会：
- 汇总到团队时间节省统计报告
- 按员工、用户故事、步骤三个维度分析
- 作为专家持续优化的依据

### 5.2 哪些步骤需要反馈

| 步骤 | 参考节省时间 | 说明 |
|------|------------|------|
| 文档整理 | 2~4 小时 | 按文档数量浮动，5个以上取上限 |
| 需求评审 | 2~3 小时 | 6 维度自动评审 |
| 生成测试点 | 3~5 小时 | 按需求复杂度浮动 |
| 生成用例 | 4~8 小时（0.5~1 人天） | 按用例数量浮动 |
| 入库知识库 | 1~2 小时 | 总结+差异对比+归档 |

> 参考时间只是建议值，你最清楚自己实际省了多少时间。

### 5.3 反馈流程（每步完成后自动触发）

**第 1 步：专家通报完成 + 询问**

```
✅ 步骤 文档整理 已完成。

📊 效能追踪（必填）：这一步为你节省了多少时间？
   参考值：约 2~4 小时（按文档数量浮动，5个以上取上限）

   👤 员工：吴香康
   📋 用户故事：___（请输入用户故事名称或编号，如 US-001-贷款审批流程优化）
   ⏱️ 节省时间：___
      - 回复"采纳"→ 使用参考值上限（4小时）
      - 或直接输入，如"4小时""0.5人天""1.5人天"

   ⚠️ 此项为必填，不可跳过。
```

**第 2 步：你回复时间**

| 你想表达的 | 你可以回复 |
|-----------|-----------|
| 采用参考值上限 | `采纳` |
| 4 小时 | `4小时` / `4个小时` / `4h` |
| 半天 | `0.5人天` / `半天` |
| 一天 | `1人天` / `一天` |
| 1.5 天 | `1.5人天` / `1.5天` |
| 3.5 小时 | `3.5` / `3.5小时` |

> 首次反馈需要填写用户故事名称（如 `US-001-贷款审批流程优化`），后续步骤会自动复用，如需更换回复"新故事：xxx"。

**第 3 步：二次确认**

专家会展示解析结果请你确认：

```
📋 请确认以下时间节省数据是否准确：

   👤 员工：吴香康
   📋 用户故事：US-001-贷款审批流程优化
   📝 步骤：文档整理（01）
   ⏱️ 节省时间：4小时（换算 4.0 小时 / 0.5 人天）
   💬 备注：原本需手动整理5个文档

   确定准确并提交？（回复"确定"保存，或修改内容）
```

**你回复**：
- `确定` / `确认` / `对` → 保存数据
- 修改内容（如"不是4小时，是3小时"）→ 专家重新解析，再次确认
- `取消` → 不保存，继续推进工作流

> **不确认不保存**。在你明确回复"确定"之前，专家不会记录任何数据。

**第 4 步：确认记录**

```
✅ 已记录：吴香康 在 US-001-贷款审批流程优化 的 文档整理 环节节省了 0.5 人天（4.0 小时）。
   📊 已同步到 Excel 文件。
```

### 5.4 数据存储说明

| 项目 | 说明 |
|------|------|
| 存储单位 | 统一为**小时**（1 人天 = 8 小时） |
| 报告展示 | 以**人天**为主展示，如"9.3 人天（74.5 小时）" |
| 本地存储 | 先写入你电脑的本地文件（兜底，即使离线也不丢数据） |
| Excel 同步 | 每天 21:00 自动同步到管理员维护的 Excel 文件 |
| GitHub 归集 | 你的数据会同步到 GitHub 仓库 `data/{你的姓名}/records.jsonl` |
| 数据安全 | 即使同步失败，本地数据不会丢失，下次同步自动补上 |

**数据流向**：

```
你反馈时间 → 二次确认 → 本地 JSONL（即时） → Excel 文件（每日21:00同步） → GitHub 仓库（归集）
```

> 你不需要关心数据最终存到哪里——只需如实反馈时间，同步全部自动完成。管理员负责维护数据归集和报告生成。

## 六、查看时间节省统计

### 6.1 怎么查看

任意时候对专家说以下任意一句：

- "查看时间节省统计"
- "查看时间统计"
- "时间节省分析"
- "效能统计"

或直接点击快捷按钮 **"查看时间节省统计"**。

专家会生成一份 HTML 报告并在对话中展示，包含：

| 模块 | 内容 |

> 所有测试人员都可以随时查看全团队统计数据，没有权限限制。

## 七、常见问题

### Q1：输入姓名后提示"不在花名册中"怎么办？

这说明你尚未被管理员授权。请联系管理员，将你的姓名添加到效贷花名册中。添加后重新打开对话即可。

注意事项：
- 输入**真实姓名**（与花名册一致），不要用昵称或简称
- 姓名前后不要加空格
- 如果确认已在花名册中但仍提示失败，检查姓名是否完全一致（如"吴香康"不能写成"吴小康"）

### Q2：专家找不到我的需求文档怎么办？

确保你提供的是**包含需求文档的目录路径**，而不是单个文件。例如：

```
✅ 帮我整理 D:\项目文件\效贷\2026Q3\贷款审批优化 下的文档
❌ 帮我整理贷款审批优化.docx
```

### Q3：XMind 文件怎么评审？

步骤④生成的 XMind 文件可以用 XMind 软件打开。你可以直接在脑图中：
- 添加新节点（补充遗漏的测试点）
- 删除节点（去掉不需要的测试点）
- 修改节点内容（调整测试点描述）
- 添加标记（标记疑问或风险）

修改保存后，直接告诉专家"XMind 评审完了"即可。

### Q4：我不想反馈时间可以吗？

时间反馈是**强制性的**，但专家不会因为你不反馈就停止工作。如果你连续拒绝，专家会记录"用户未反馈"并继续推进。

不过我们强烈建议你如实反馈——这些数据是衡量专家价值、争取团队持续投入的关键依据。

### Q5：时间反馈填错了怎么办？

在二次确认环节，你可以直接说"不是 X 小时，是 Y 小时"来修改。如果已经保存后发现错误，联系管理员在 Excel 文件中手动修正即可。

### Q6：我可以只做某一步吗？

可以。这就是**模式 B（单步模式）**。例如：
- "我有整理好的需求 MD，帮我评审" → 只执行步骤②
- "我有评审后的 XMind，帮我生成用例" → 只执行步骤⑥

### Q7：步骤之间会自动跳转吗？

**不会**。每完成一个步骤，专家会停下来等你确认。你说"继续"或给出下一步指令才会推进。这是为了确保你有充分的检查和调整时间。

### Q8：入库知识库是必须的吗？

**不是**。步骤⑦是可选的。只有你说"入库"时才会执行。如果你觉得这次需求没有值得沉淀的内容，跳过即可。

### Q9：专家会记住我上次做到哪了吗？

每次新开对话，专家会从头开始。如果你上次做到某个步骤，这次想继续，可以直接告诉专家：
- "我之前已经整理好了文档，帮我评审"
- 然后提供上次的产出文件路径

### Q10：报告里的"人天"和"小时"怎么换算？

1 人天 = 8 小时。无论你输入的是小时还是人天，底层存储统一为小时，报告展示以人天为主。例如：
- 输入"4小时" → 存 4.0 小时 → 报告显示"0.5 人天（4.0 小时）"
- 输入"1.5人天" → 存 12.0 小时 → 报告显示"1.5 人天（12.0 小时）"

### Q11：其他业务线的同事能用这个专家吗？

**不能**。效贷测试专家专用于效贷业务线，花名册中只有效贷测试团队的人员。其他业务线（如泾渭云、智慧记）需要单独搭建各自的专家。

### Q12：PAT 是什么？为什么需要它？

PAT（Personal Access Token）是 GitHub 的个人访问令牌，类似于密码。你的时间节省数据需要同步到 GitHub 仓库中进行团队归集，PAT 就是让专家有权限替你推送数据的"钥匙"。

- 每人一个，互不影响
- 只有 `Contents: Read and Write` 权限，仅限本仓库
- 不会暴露你的 GitHub 密码
- 如果不小心泄露，到 GitHub 设置中撤销重新生成即可

### Q13：专家更新了怎么办？我需要重新安装吗？

当管理员推送新版本到 GitHub 仓库后，WorkBuddy 会自动检测更新。你可能会看到更新提示，点击更新即可——不需要重新配置 PAT 或重新设置任何东西。

如果没看到更新提示但管理员通知了新版本，可以尝试：
1. 关闭当前对话，重新打开专家
2. 如果仍无更新，在 WorkBuddy 团队市场设置中手动刷新

### Q14：我的工时数据会暴露给其他人吗？

**不会**。你的工时数据存储在 `data/{你的姓名}/records.jsonl` 中。虽然仓库是公开的，但：

- 数据内容只有时间节省记录（姓名、步骤、小时数），不含敏感业务信息
- 只有被邀请为 Collaborator 的人才能推送数据
- 管理员可以查看全团队数据用于汇总报告
- 其他测试人员可以通过"查看时间节省统计"看到团队汇总数据（这是设计如此，用于团队透明度）

如果你不希望自己的数据被其他同事看到，请联系管理员讨论隐私方案。

### Q15：换电脑了怎么办？

换电脑后需要重新：
1. 安装 WorkBuddy
2. 添加团队市场仓库地址
3. 安装效贷测试专家
4. 重新提供你的 PAT（如果在旧电脑上没有导出）

你的历史数据不会丢失——它们已经存储在 GitHub 仓库和 Excel 文件中。

### Q16：Confluence 页面提取失败怎么办？

常见原因及处理方法：

| 现象 | 原因 | 处理方法 |
|------|------|---------|
| 提示缺少环境变量 | 未配置 Confluence 连接信息 | 按专家提示配置 `CONFLUENCE_URL`、`CONFLUENCE_USERNAME`、`CONFLUENCE_PASSWORD` 后重试 |
| 提示无法连接服务器 | `CONFLUENCE_URL` 不正确或网络不通 | 检查 URL 是否正确，确认可访问内网 Confluence |
| 提示认证失败（401） | 用户名或密码/Token 错误 | 检查 `CONFLUENCE_USERNAME` 和 `CONFLUENCE_PASSWORD` |
| 提示页面 404/403 | 页面不存在或无权限 | 检查 URL 是否正确，或联系 Confluence 管理员开通权限 |
| 图片未识别 | 未配置 Claude API Key | 不影响文字提取，仅跳过图片 AI 描述。可后续手动补充描述 |

> Confluence 环境变量只需配置一次，配置成功后同一台电脑后续提取不再需要重复配置。

### Q17：Confluence 提取和本地目录整理有什么区别？

| 对比项 | 本地目录整理 | Confluence 页面提取 |
|--------|-------------|-------------------|
| 输入 | 本地文件夹路径 | Confluence URL |
| 处理内容 | Word/PDF/图片/Excel 等本地文件 | Confluence 页面正文 + 图片/附件 |
| 输出 | `{目录名}_整理版_v{version}.md` | `{页面标题}_整理版_v1.0.md` |
| 归档 | 复制文件到 `source/` | 下载图片到 `source/images/` |
| 后续 | 都可以进入步骤②需求评审 | 都可以进入步骤②需求评审 |
