# 需求评审 Prompt

> 此 Prompt 用于 AI 对需求文档和技术方案进行专业评审

---

## ⛔ 禁止 AI 擅自做决定（强制按流程规范执行）

> **⚠️ 执行此 Prompt 前，必须完整阅读本文档！禁止凭记忆或推断执行！**

| 禁止项 | 说明 | 正确做法 |
|--------|------|---------|
| ⛔ 禁止假设已读取 | 不假设前置步骤已完成，必须实际执行每步读取操作 | 每个步骤按规范重新读取，不假设前置状态 |
| ⛔ 禁止简化 Todo | Todo 清单必须完整，不可省略强制检查项（⭐标记项） | 按规范创建完整 Todo 清单（13项），逐项标记状态 |
| ⛔ 禁止跳过步骤 | 强制项（⭐⭐⭐）不可跳过，必须逐项执行并标记状态 | 强制项必须执行，AI 无权判断跳过 |
| ⛔ 禁止自行判断 | AI 不可自行判断「不重要」「已完成」「无需执行」 | 按流程规范执行，不做任何擅自决策 |
| ⛔ 禁止假设状态 | 标记「已完成」前必须实际执行该操作，不可凭记忆标记 | 每项必须实际执行后才能标记 completed |
| ⛔ 禁止跳过前置检查 | 前置检查（0-1/0-2/0-3）不可跳过 | 必须先执行前置检查，再执行评审流程 |

**典型违规场景**：

| 违规行为 | 正确做法 |
|---------|---------|
| 假设"步骤②已读过需求文档，无需再读" | 每个步骤按规范重新读取，不假设前置状态 |
| 简化 Todo 为 5 项而非 13 项 | 按规范创建完整 Todo 清单，逐项标记状态 |
| 标记"已完成"但实际未执行读取操作 | 每项必须实际执行后才能标记 completed |
| 跳过"扫描图片目录"认为不重要 | 强制项必须执行，AI 无权判断跳过 |

---

## ⚠️ 前置检查流程（最高优先级）⭐⭐⭐

> **AI 执行需求评审前，必须先完成前置检查！跳过前置检查属于流程错误！**
> 
> 💡 **项目根目录已缓存时直接使用，无需重复定位和检查初始化。**

### 前置检查步骤

| 序号 | 检查项 | 强制 | AI 行为 |
|------|--------|------|---------|
| 0-1 | 项目根目录（如未缓存则定位） | ⭐⭐⭐ | 已缓存→直接使用；未缓存→向上查找 `.skill/`（最多3层） |
| 0-2 | 项目初始化检查（仅首次定位时执行） | ⭐⭐⭐ | 未初始化→创建知识库；已初始化→跳过 |
| 0-3 | 整理版 MD 检查 | ⭐⭐⭐ | 检查是否存在 `*整理版*.md`（来源可以是本地目录整理或 Confluence 页面提取），**不存在则提示先执行①文档整理/Confluence 提取** |

### 检查流程

```
用户触发需求评审
    │
    ↓
检查 0-1：项目根目录
    │
    ├─ 已缓存 → 直接使用
    ├─ 未缓存 → 定位（向上查找 .skill/，最多3层）→ 缓存
    │   └─ 定位失败 → 询问用户「项目根目录在哪里？」
    │
    ↓
检查 0-2：项目初始化（仅首次定位时执行）
    │
    ├─ 已初始化 → 继续
    ├─ 未初始化 → 执行初始化流程（创建目录+询问配置）
    │
    ↓
检查 0-3：整理版 MD 是否存在
    │
    ├─ 存在 → 读取整理版 MD，继续评审
    │        （来源不限：本地目录整理 或 Confluence 页面提取 均可）
    ├─ 不存在 → ⚠️ 提示用户先执行①文档整理，或提供 Confluence URL 提取页面
    │
    ↓
创建 Todo 清单 → 执行评审流程
```

### ❌ 禁止行为

| 禁止项 | 说明 |
|--------|------|
| ❌ 跳过前置检查 | 不执行前置检查直接开始评审属于流程错误 |
| ❌ 每个步骤重复定位项目根目录 | 应使用缓存的项目根目录 |
| ❌ 每个步骤重复检查初始化 | 仅首次定位时检查 |
| ❌ 整理版 MD 不存在时强行评审 | 必须先完成文档整理（本地目录）或 Confluence 页面提取 |

## ⚠️ 执行前必须创建 Todo 清单 ⭐

> **AI 执行此流程时，必须先调用 `todo_write` 创建以下 todo 清单，逐项完成并标记状态！**
>
> ⚠️ **注意：Todo 清单前 3 项为前置检查步骤，必须先完成！**

**步骤②子 Todo 清单（13项强制步骤）**：

| 序号 | 步骤 | 强制 | 输出/验证 |
|------|------|------|----------|
| 0-1 | 项目根目录（如未缓存则定位） | ⭐⭐⭐ | 已缓存→直接使用；未缓存→定位并缓存 |
| 0-2 | 项目初始化检查（仅首次定位时执行） | ⭐⭐⭐ | 未初始化→创建；已初始化→跳过 |
| 0-3 | 整理版 MD 检查 | ⭐⭐⭐ | 检查 `*整理版*.md`，不存在则提示先执行① |
| 1 | 读取 JSON 格式规范（本文档「JSON 格式规范」章节） | ⭐ 强制 | 理解 doc_title、categories[].items[] 等关键字段 |
| 2 | 读取精华库 {kb_path}/patterns/精华库/common_root_causes.md + common_omissions.md | ⭐ 默认执行 | 轻量路径：AI 错题本（≤300 Token） |
| 3 | 读取知识库历史文档 {kb_path}/requirements/*.md | 可选 | **完整路径**：用户说"结合历史"等关键词时执行 |
| 4 | 读取需求文档 MD | ⭐ 强制 | 主评审文档 |
| 5 | 搜索技术方案 MD（搜索 TR_*.md） | 推荐 | 关联技术方案 |
| 6 | 扫描图片目录 images/ | ⭐ 强制 | 记录到 document_status.images |
| 7 | 检查附件目录 attachments/ | ⭐ 强制 | 记录到 document_status.attachments |
| 8 | 多维度评审（按本文档定义的评审维度） | ⭐ 强制 | 逐维度检查 |
| 9 | 需求-技术方案对比分析 | ⭐ 强制 | 记录到 comparison.diff_items |
| 10 | 技术风险识别 | ⭐ 强制 | 记录到 tech_risks |
| 11 | 按格式规范生成 JSON | ⭐ 强制 | 逐字段对照验证 |
| 12 | 调用脚本生成评审报告 MD（generate_review_report.py） | ⭐ 强制 | 包含知识库读取状态、图片解析状态章节 |

**AI 执行流程**：
1. 收到步骤②指令 → 立即调用 `todo_write` 创建 13 项子清单（含前置检查）
2. **先执行 0-1、0-2、0-3 前置检查步骤**
3. 前置检查完成后 → 逐项执行 1-12 评审步骤
4. 每完成一项立即标记 `completed`
5. 强制项跳过 → 用户会看到 todo 状态未完成 → 立即发现问题

---

## 任务目标

对需求文档和技术方案文档（MD 格式）进行专业评审，从多个维度发现潜在问题，生成结构化评审报告，用于需求评审会议、技术方案评审会议或文档完善。

---

## 输入

| 输入项 | 来源 | 必选 |
|--------|------|------|
| 需求文档 MD | 用户提供 | ⭐ 必选 |
| 技术方案 MD | 自动搜索 | 推荐 |
| 图片目录 | 同目录 `images/` | 推荐 |
| 知识库历史文档 | `{kb_path}/requirements/` | 推荐 |
| 精华库 | `{kb_path}/patterns/精华库/` | 推荐 |

---

## 输出

> ⚠️ **输出路径规则**：所有输出文件统一存放到用户故事工作目录（`D:\效贷-产品需求\{用户故事编号}-{用户故事名称}`），由步骤①确认并缓存为 `{user_story_dir}`。如会话中已缓存用户故事目录路径，直接使用；如未缓存，需先向测试人员确认用户故事信息（编号+名称）并定位/创建工作目录。

| 输出项 | 格式 | 命名 | 输出路径 |
|--------|------|------|---------|
| 评审报告 | .md | `{需求名}_评审报告.md` | `{user_story_dir}/` |
| 评审数据 | .json | `{需求名}_评审数据.json` | `{user_story_dir}/` |

## 字段说明（必读）⭐

### 顶级字段

| 字段路径 | 含义 | 必填 | 说明 |
|---------|------|------|------|
| `doc_title` | 文档名称 | ✅ | 格式：`{需求名称}` |
| `doc_type` | 文档类型 | ✅ | `需求文档`/`技术方案`/`混合文档` |
| `review_date` | 评审日期 | ✅ | YYYY-MM-DD 格式 |
| `reviewer` | 评审人 | ✅ | 固定值：`AI 评审` |
| `summary` | 评审概览 | ✅ | 问题统计 |
| `document_status` | 文档读取状态 | ✅ | **8 个必选状态标记** |
| `categories` | 问题分类数组 | ✅ | **顶层数组，所有问题放在此处** |
| `comparison` | 需求-技术方案对比 | ⚠️ | 有技术方案时必填 |
| `historical_comparison` | 历史对比分析 | ⚠️ | 有历史记录时必填 |
| `tech_risks` | 技术风险提示 | ✅ | 数组格式 |
| `completeness_check` | 完整性检查 | ✅ | 数组格式 |

### document_status 字段（8 个必选状态标记）

| 字段路径 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| `document_status.knowledge_base` | 对象 | ✅ | 知识库读取状态 |
| `knowledge_base.exists` | boolean | ✅ | **知识库目录是否存在**（true=目录存在，false=目录不存在/未初始化） |
| `knowledge_base.message` | string | ✅ | 读取结果描述（如："暂无历史文档"） |
| `knowledge_base.history_items` | 数组 | ⚠️ | 历史文档列表（有数据则填写，无数据则空数组） |
| `document_status.related_documents` | **数组** | ✅ | **⚠️ 必须是数组，不可是字符串！** |
| `related_documents[].type` | string | ✅ | 文档类型 |
| `related_documents[].filename` | string | ✅ | 文件名 |
| `related_documents[].status` | string | ✅ | 读取状态 |
| `related_documents[].remark` | string | ✅ | 备注 |
| `document_status.images` | **数组** | ✅ | **⚠️ 必须是数组，不可是字符串！** |
| `images[].name` | string | ✅ | 图片名称 |
| `images[].location` | string | ✅ | 引用位置 |
| `images[].status` | string | ✅ | 解析状态（✅ 已解析/❌ 未解析） |
| `images[].type` | string | ✅ | 内容类型（流程图/界面原型图/状态机图等） |
| `images[].info` | string | ⚠️ | 关键信息（已解析时填写） |
| `images[].reason` | string | ⚠️ | **未解析原因**（未解析时必填，说明原因和解决建议） |
| `document_status.attachments` | 数组 | ⚠️ | 附件列表（有则填写） |

> ⚠️ **知识库状态语义说明**：
> - `exists=true` + `history_items=[]` → 知识库已初始化，但暂无历史数据
> - `exists=true` + `history_items有数据` → 正常读取历史文档
> - `exists=false` → 知识库目录不存在（项目未初始化）

> ⚠️ **图片未解析原因规范**：
> - 必须说明具体原因（如：当前模型不支持图片输入）
> - 必须提供解决建议（如：切换模型或手动描述内容）
> - 必须标注影响范围（如：流程图、界面原型关键信息遗漏）

### categories 字段（问题分类）

| 字段路径 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| `categories` | **数组** | ✅ | **⚠️ 顶层数组，不嵌套在 requirement_review/technical_review 下！** |
| `categories[].name` | string | ✅ | 分类名称（如：需求疑问、数据库设计） |
| `categories[].items` | 数组 | ✅ | 问题项数组 |
| `items[].title` | string | ✅ | 问题标题（简短概括） |
| `items[].location` | string | ✅ | 问题位置 |
| `items[].description` | string | ✅ | 问题描述 |
| `items[].suggestion` | string | ✅ | 改进建议 |
| `items[].severity` | string | ✅ | `critical`/`major`/`minor` |
| `items[].status` | string | ✅ | 处理状态（待解决/缺失/待确认等） |

### comparison 字段（需求-技术方案对比）

| 字段路径 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| `comparison.diff_items` | 数组 | ⚠️ | 差异项列表 |

### 步骤 3：逐维度评审 ⭐

> ⚠️ **评审聚焦原则**：聚焦需求本身的缺陷，不评审运维细节、实现细节！

---

#### 一、需求评审维度（必审）

| 维度 | 检查项 | 示例 |
|------|--------|------|
| **1. 需求疑问** | 业务逻辑不清、边界缺失、状态流转缺失、异常处理缺失 | "当天有效"是几点到几点？ |
| **2. 模糊点** | 用词模糊、量化缺失、条件不完整、时间定义模糊 | "快速返回结果"—多快算快速？ |
| **3. 历史冲突** | 流程冲突、状态冲突、数据冲突、权限冲突 | 新流程与现有流程矛盾 |
| **4. 完整性检查** | 版本记录、需求背景、核心功能覆盖 | 是否有版本变更记录 |
| **5. 可测试性** | 验证标准明确、测试数据可得、边界值可测 | "提升用户体验"无法量化测试 |
| **6. 需求风险** | 第三方依赖风险（需定义超时/降级）、合规风险 | 依赖外部系统稳定性 |

---

#### 二、技术方案评审维度（精简）

> ⚠️ **只评审影响需求实现的关键技术问题，不评审运维细节！**

| 维度 | 检查项 | 示例 | 评审深度 |
|------|--------|------|---------|
| **7. 核心流程** | 流程完整、状态流转、关键异常分支 | 是否遗漏关键异常分支 | **必审** |
| **8. 数据库设计** | 关键字段类型、必要字段缺失 | 金额字段是否用 DECIMAL | **必审** |
| **9. 接口设计** | 关键参数缺失、幂等设计 | 是否缺少关键返回字段 | **必审** |
| **10. 需求-技术方案一致性** | 技术方案是否覆盖需求功能、状态定义一致 | 需求15天 vs 技术14天 | **必审** |
| **11. 安全合规提示** | 敏感数据加密提示、权限控制提示 | 只提示"需加密"，不指定算法 | **轻量提示** |
| **12. 第三方依赖风险** | 外部系统超时、降级、重试机制 | 需定义超时阈值 | **必审** |

---

#### 三、不评审的内容 ⭐ 明确约束

| 类别 | 不评审内容 | 原因 |
|------|-----------|------|
| **运维细节** | 灰度发布方案、故障恢复方案、日志格式规范、日志存储策略 | 这是运维设计阶段的内容 |
| **监控细节** | 告警渠道、告警去重、告警恢复、监控指标定义 | 需求只需说"异常需告警"，具体配置是运维决定 |
| **实现细节** | 具体加密算法（AES-256）、具体缓存策略、具体日志级别 | 需求只需说"需加密"，具体方案是技术决定 |
| **扩展细节** | 多资方扩展架构、插件化设计、水平扩展方案 | 这是架构设计阶段的内容，除非需求明确要求 |
| **性能细节** | 具体轮询频率、具体缓存时效、批量处理数量上限 | 需求只需说"需考虑性能"，具体参数是技术决定 |
| **技术选型** | 具体框架版本、具体组件选型 | 这是技术团队决定，除非涉及新技术风险 |

#### 四、评审问题分级标准

| 级别 | 定义 | 适用场景 |
|------|------|---------|
| 🔴 **严重** | 阻塞需求实现、业务逻辑错误、状态流转缺失 | 必须在评审会议讨论 |
| 🟡 **重要** | 影响测试验证、需求与技术方案不一致、边界条件不明 | 开发前补充 |
| 🟢 **一般** | 提示类问题、建议优化、非阻塞项 | 可后续迭代 |

> ⚠️ **杜绝提出"一般"级别的运维细节问题！**

| `document_status.attachments` | 附件解析状态 |
| `comparison.diff_items` | 需求-技术方案对比分析 |
| `historical_comparison.items` | 历史对比分析 |
| `tech_risks` | 技术风险提示 |
| `items[].status` | 问题处理状态标记 |

---

### 步骤 6：调用脚本生成 MD 报告 ⭐

> ⚠️ **必须调用脚本生成 MD 报告，不可手动 write_file！**

**6.1 调用命令**

```bash
python {技能目录}/scripts/generate_review_report.py --input <评审JSON路径> --output <评审报告MD路径>
```

**6.2 参数说明**

| 参数 | 说明 | 示例 |
|------|------|------|
| `--input` | 评审 JSON 文件路径 | `E:\项目\需求_评审数据.json` |
| `--output` | 输出 MD 报告路径 | `E:\项目\需求_评审报告.md` |

**6.3 输出验证**

脚本执行后，验证以下 8 个必选状态标记是否包含在报告中：

| 序号 | 状态标记 | 验证方式 |
|------|---------|---------|
| 1 | 知识库读取状态 | 检查「知识库读取状态」章节 |
| 2 | 关联文档读取状态 | 检查「关联文档读取状态」章节 |
| 3 | 图片解析状态 | 检查「图片解析状态」章节 |
| 4 | 附件解析状态 | 检查「附件解析状态」章节（如有附件） |
| 5 | 需求-技术方案对比分析 | 检查对比分析章节（如有技术方案） |
| 6 | 历史对比分析 | 检查历史对比章节（如有历史记录） |
| 7 | 技术风险提示 | 检查技术风险章节 |
| 8 | 问题处理状态标记 | 检查每个问题是否包含「处理状态」 |

**6.4 流程总结**

```
AI 多维度分析 → 生成评审 JSON 数据 → 调用脚本 → 输出 MD 评审报告
      │                │                  │                │
   (步骤1-4)        (步骤5)           (步骤6)          (输出验证)
```

## 使用示例

### 基础用法（单文件评审）

```
用户：帮我评审 E:\项目\额度测算.md 这个需求文档

AI 执行：
1. 读取需求文档
2. 自动识别文档类型
3. 搜索知识库历史文档
4. 搜索关联技术方案（如有）
5. 解析图片（如有）
6. 按对应维度评审
7. 生成评审报告
```

### 技术方案评审

```
用户：帮我评审 E:\项目\TR_额度测算.md 这个技术方案

AI 执行：
1. 读取技术方案文档
2. 搜索关联需求文档
3. 搜索需求评审报告
4. 对比分析需求-技术方案
5. 按技术方案维度评审
6. 生成评审报告
```

### 带历史对比

```
用户：用 E:\项目\额度测算.md 和 E:\项目\历史\额度测算_v1.0.md 对比评审

AI 执行：
1. 读取两个文档
2. 对比分析差异
3. 识别与历史逻辑冲突点
4. 生成评审报告
```

## 注意事项

1. **评审结果需人工确认**：AI 评审作为参考，需产品/架构师确认
2. **技术方案必须关联需求**：不可独立评审
3. **图片必须解析**：流程图、状态图是关键信息
4. **历史对比必须执行**：知识库历史文档不可跳过
5. **输出位置**：评审报告输出到被评审 MD 文件所在目录
6. **严重级别定义**：阻塞=严重、影响=重要、优化=一般
