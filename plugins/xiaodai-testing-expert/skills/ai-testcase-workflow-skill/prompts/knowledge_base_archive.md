# 入库知识库执行规则

> 本文档定义步骤⑦入库知识库的详细执行规则
>
> **AI 必须在执行入库前阅读此文档！**

---

## ⛔ 禁止 AI 擅自做决定（强制按流程规范执行）

> **⚠️ 执行此 Prompt 前，必须完整阅读本文档！禁止凭记忆或推断执行！**

| 禁止项 | 说明 | 正确做法 |
|--------|------|---------|
| ⛔ 禁止假设已读取 | 不假设前置步骤已完成，必须实际执行每步读取操作 | 每个步骤按规范重新读取，不假设前置状态 |
| ⛔ 禁止简化 Todo | Todo 清单必须完整，不可省略强制检查项（⭐标记项） | 按规范创建完整 Todo 清单（12项），逐项标记状态 |
| ⛔ 禁止跳过步骤 | 强制项（⭐）不可跳过，必须逐项执行并标记状态 | 强制项必须执行，AI 无权判断跳过 |
| ⛔ 禁止自行判断 | AI 不可自行判断「不重要」「已完成」「无需执行」 | 按流程规范执行，不做任何擅自决策 |
| ⛔ 禁止假设状态 | 标记「已完成」前必须实际执行该操作，不可凭记忆标记 | 每项必须实际执行后才能标记 completed |
| ⛔ 禁止跳过重复入库检查 | 重复入库检查不可跳过 | 必须检查知识库是否已存在同名文档 |
| ⛔ 禁止跳过差异对比 | 测试点 JSON 对比分析不可跳过 | 必须对比评审前后差异，反思遗漏 |
| ⛔ 禁止跳过质量分析 | 测试用例生成质量分析不可跳过 | 必须检查前置条件、步骤细化、数据实例化 |
| ⛔ 禁止跳过进化日志更新 | 更新进化日志 EVOLUTION_LOG.md 不可跳过 | 必须追加入库记录和进化指标 |
| ⛔ 禁止跳过精华库提炼 | 自动提炼精华到 patterns/ 不可跳过 | 必须提炼高频根因到精华库 |

**典型违规场景**：

| 违规行为 | 正确做法 |
|---------|---------|
| 假设"文件已入库，跳过检查" | 必须实际检查知识库目录，确认是否已存在 |
| 简化 Todo 为 5 项而非 12 项 | 按规范创建完整 Todo 清单，逐项标记状态 |
| 标记"已完成"但实际未执行读取操作 | 每项必须实际执行后才能标记 completed |
| 跳过"差异对比分析"认为不重要 | 必须对比评审前后差异，反思 AI 遗漏原因 |
| 跳过"进化日志更新"认为已完成 | 必须追加本次入库记录到进化日志 |
| 跳过"精华库提炼"认为不重要 | 必须提炼高频根因到精华库，固定 Top 10 上限 |

## ⭐ 知识库命名规范（避免重名）

> **核心原则**：使用「需求编号_需求名称_版本日期」三段式命名，确保唯一性

### 命名格式

```
格式：{需求编号}_{需求名称}_{版本日期}
```

| 字段 | 说明 | 来源 | 示例 |
|------|------|------|------|
| 需求编号 | 用户故事编码 | relateReqCode 参数 | `PRJ-XXXXXXX` |
| 需求名称 | 简化的需求名称 | 原文档名简化 | `功能名称` |
| 版本日期 | 需求版本日期 | version 参数解析 | `YYYYMMDD` |

### 各目录文件命名规则

| 目录 | 命名格式 | 说明 |
|------|---------|------|
| **requirements/** | `{编号}_{名称}_{日期}.md` | 需求文档 |
| **review-reports/** | `{编号}_{名称}_{日期}_评审报告.md` | 评审报告 |
| **tech-solutions/** | `TR_{编号}_{名称}_{日期}.md` | 技术方案 |
| **testcases/** | `{编号}_{名称}_{日期}_测试点.json` | 测试点 JSON |
| **testcases/** | `{编号}_{名称}_{日期}_测试用例.json` | 测试用例 JSON |
| **testcases/** | `{编号}_{名称}_{日期}_测试用例.xlsx` | 测试用例 Excel |

### 无需求编号时的处理

如无需求编号（relateReqCode 为空），使用「需求名称_版本日期」二段式命名：

| 目录 | 命名格式 |
|------|---------|------|
| requirements/ | `{名称}_{日期}.md` |
| review-reports/ | `{名称}_{日期}_评审报告.md` |
| tech-solutions/ | `TR_{名称}_{日期}.md` |
| testcases/ | `{名称}_{日期}_测试点.json` |
| testcases/ | `{名称}_{日期}_测试用例.json` |

### 命名生成逻辑

```
命名规则：
  1. 从版本号提取日期：VYYYY.MM.DD → YYYYMMDD（去除 V 和点）
  2. 简化需求名称：去除特殊字符，保留核心关键词
  3. 组合命名：
     - 有编号：{编号}_{名称}_{日期}
     - 无编号：{名称}_{日期}
```

## 🔍 前置检查

### 步骤 1：读取配置和定位知识库

```
1. 从项目根目录读取 .skill/project.yaml
2. 获取知识库路径：
   - 优先使用 absolute_path（共享知识库）
   - 否则使用 {项目根目录}/{relative_path}
3. 确认知识库目录存在
4. 确认知识库子目录结构完整：
   - requirements/
   - review-reports/
   - tech-solutions/
   - testcases/
   - patterns/
```

### 步骤 2：重复入库检查 ⭐

> **入库前必须检查知识库是否已存在同名文档，避免重复入库！**
>
> ⚠️ 使用统一命名规范检查：`{编号}_{名称}_{日期}`

**检查内容**：

| 检查项 | 检查路径 | 已存在时处理 |
|--------|---------|-------------|
| 需求文档 | `{kb_path}/requirements/{编号}_{名称}_{日期}.md` | ✅ 已入库，跳过 |
| 评审报告 | `{kb_path}/review-reports/{编号}_{名称}_{日期}_评审报告.md` | ✅ 已入库，跳过 |
| 技术方案 | `{kb_path}/tech-solutions/TR_{编号}_{名称}_{日期}.md` | ✅ 已入库，跳过 |
| 测试点 JSON | `{kb_path}/testcases/{编号}_{名称}_{日期}_测试点.json` | ✅ 已入库，跳过 |
| 测试用例 JSON | `{kb_path}/testcases/{编号}_{名称}_{日期}_测试用例.json` | ✅ 已入库，跳过 |

**检查结果处理**：

| 状态 | 处理方式 |
|------|---------|
| 全部已存在 | 输出「所有文档已入库，无需重复操作！流程完成。」 |
| 部分缺失 | 只入库缺失的文档 |
| 全部缺失 | 执行完整入库流程 |

**输出检查结果表格**：

| 目录 | 文件名 | 状态 | 操作 |
|------|--------|------|------|
| requirements/ | {编号}_{名称}_{日期}.md | ✅ 已存在 / ❌ 缺失 | 跳过 / 入库 |
| review-reports/ | {编号}_{名称}_{日期}_评审报告.md | ✅ 已存在 / ❌ 缺失 | 跳过 / 入库 |
| tech-solutions/ | TR_{编号}_{名称}_{日期}.md | ✅ 已存在 / ❌ 缺失 | 跳过 / 入库 |
| testcases/ | {编号}_{名称}_{日期}_测试点.json | ✅ 已存在 / ❌ 缺失 | 跳过 / 入库 |
| testcases/ | {编号}_{名称}_{日期}_测试用例.json | ✅ 已存在 / ❌ 缺失 | 跳过 / 入库 |

### 步骤 5：测试点 JSON 对比分析 ⭐

### 步骤 6：测试用例生成质量分析 ⭐

> **测试用例是最终产物，必须分析生成质量！**

#### 6.1 用例细化质量检查

| 检查项 | 检查方式 | 质量评估 |
|--------|---------|---------|
| **前置条件完整性** | 检查每个用例 preCondition 是否非空 | 缺失率统计 |
| **步骤细化程度** | 检查步骤是否具体可执行（>=2个） | 抽象占位符检出率 |
| **步骤编号规范** | 检查是否使用「1、」格式 | 错误格式检出率 |
| **预期结果可验证** | 检查 output 是否可量化验证 | 模糊描述检出率 |
| **数据实例化** | 检查是否使用具体数据而非占位符 | 占位符检出率 |
| **步骤与预期对应** | 检查 input/output 数量是否相等 | 不匹配检出率 |

#### 6.2 分类占比合理性检查

| 分类 | 目标占比 | 检查方式 | 偏差评估 |
|------|---------|---------|---------|
| 功能测试 | 60%~70% | functional / total_cases | ✅ 合理 / ⚠️ 偏高 / ⚠️ 偏低 |
| 集成测试 | 20%~30% | integration / total_cases | ✅ 合理 / ⚠️ 偏低，需加强跨模块用例 |
| 安全测试 | 5%~10% | security / total_cases | ✅ 合理 / ⚠️ 缺失安全场景 |
| 性能测试 | 按需 | performance / total_cases | ✅ 合理 / ⚠️ 遗漏性能要求 |

#### 6.3 用例细化遗漏分析

常见细化遗漏场景：

| 遗漏类型 | 说明 | 检出标准 |
|---------|------|---------|
| **前置条件过于简单** | 只写"登录系统"，缺少具体数据状态 | 长度 < 20 字符 |
| **步骤不够具体** | 写"查看字段"，未写查看哪个字段 | 含模糊词：查看、验证、检查 |
| **预期结果模糊** | 写"校验通过"，未写具体校验结果 | 含模糊词：通过、正确、成功 |
| **数据使用占位符** | 写"某客户"，未写具体客户编码 | 含占位词：某、任意、某个 |
| **步骤编号错误** | 使用"1. "而非"1、" | 正则匹配 `\d\.` |
| **步骤数量不足** | 步骤只有 1 个 | input 换行数 < 2 |
