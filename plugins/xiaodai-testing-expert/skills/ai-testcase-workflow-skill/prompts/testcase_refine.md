## ⛔ 禁止 AI 擅自做决定（强制按流程规范执行）

> **⚠️ 执行此 Prompt 前，必须完整阅读本文档！禁止凭记忆或推断执行！**

| 禁止项 | 说明 | 正确做法 |
|--------|------|---------|
| ⛔ 禁止假设已读取 | 不假设前置步骤已完成，必须实际执行每步读取操作 | 每个步骤按规范重新读取，不假设前置状态 |
| ⛔ 禁止简化 Todo | Todo 清单必须完整，不可省略强制检查项（⭐标记项） | 按规范创建完整 Todo 清单（10项），逐项标记状态 |
| ⛔ 禁止跳过步骤 | 强制项（⭐）不可跳过，必须逐项执行并标记状态 | 强制项必须执行，AI 无权判断跳过 |
| ⛔ 禁止自行判断 | AI 不可自行判断「不重要」「已完成」「无需执行」 | 按流程规范执行，不做任何擅自决策 |
| ⛔ 禁止假设状态 | 标记「已完成」前必须实际执行该操作，不可凭记忆标记 | 每项必须实际执行后才能标记 completed |
| ⛔ 禁止跳过 XMind 解析 | 调用脚本解析评审后 XMind 不可跳过 | 必须调用 parse_xmind.py 解析 XMind |
| ⛔ 禁止跳过整理版 MD 读取 | 如存在整理版 MD 必须读取 | 必须读取以补充业务上下文 |
| ⛔ 禁止跳过技术方案读取 | 如存在技术方案必须读取 | 必须读取以补充接口定义 |

**典型违规场景**：

| 违规行为 | 正确做法 |
|---------|---------|
| 假设"XMind已解析，直接使用JSON" | 步骤⑥必须调用脚本解析 XMind，获取唯一测试点来源 |
| 简化 Todo 为 5 项而非 10 项 | 按规范创建完整 Todo 清单，逐项标记状态 |
| 标记"已完成"但实际未执行读取操作 | 每项必须实际执行后才能标记 completed |
| 整理版 MD 存在但不读取 | 必须读取整理版 MD 补充业务规则、边界条件 |
| 技术方案存在但不读取 | 必须读取技术方案补充接口定义、字段说明 |

> **AI 执行此流程时，必须先调用 `todo_write` 创建以下 todo 清单，逐项完成并标记状态！**

**步骤⑥子 Todo 清单（9项强制步骤）**：

| 序号 | 步骤 | 强制 | 输出/验证 |
|------|------|------|----------|
| 1 | 读取 JSON 格式规范（本文档「JSON 格式规范」章节） | ⭐ 强制 | 理解 testcases[].id、preCondition、input、output 等关键字段 |
| 2 | 调用脚本解析评审后的 XMind | ⭐ **强制** | **唯一测试点来源！调用 parse_xmind.py 解析。文件名不限，检查内容是否包含评审标记** |

| 5 | 读取整理版 MD（如存在） |  强制（如存在） | **一份文件包含所有内容**：需求文档 + 技术方案 + 图片解析 + 附件 |

**AI 执行流程**：
1. 收到步骤⑥指令 → 立即调用 `todo_write` 创建 9 项子清单
2. 逐项执行 → 每完成一项立即标记 `completed`
3. 强制项跳过 → 用户会看到 todo 状态未完成 → 立即发现问题

**⚠️ 参数来源说明**：
- team/product/modulePath → 从 `{项目根}/.skill/project.yaml` 读取，**不需要用户输入**

## 任务目标

将**评审后**的 XMind 测试点扩展为详细测试用例，输出 Excel 格式用于测试平台导入。AI 自动将测试点细化为具体步骤，生成测试数据，确保用例可执行、可验证。

---

## 输入

| 输入项 | 来源 | 必选 | 说明 |
|--------|------|------|------|
| **评审后的 XMind** | 用户提供（直接在 AI 生成的原始 XMind 上添加颜色标记，无需重命名） | ⭐ **必选** | **唯一测试点来源**，调用脚本解析为 JSON |
| 测试用例参数 | 用户问答收集 | ⭐ 必选 | caseGroup/version/manager/relateReqCode |
| 项目配置 | {项目根}/.skill/project.yaml | ⭐ **必选** | team/product/modulePath 从配置读取 |
| 整理版 MD | 同目录 `{需求名}_整理版*.md` | ⭐ 如存在 | 补充业务规则、边界条件、业务上下文 |
| 技术方案 | 同目录 `TR_*.md` | 推荐 如存在 | 补充接口定义、字段说明、数据结构 |

> 💡 **整理版 MD 存在时必须读取**，结合 MD 分析推导用例；不存在时仅凭 XMind 细化

---

## 输出

> ⚠️ **输出路径规则**：所有输出文件统一存放到用户故事工作目录（`D:\效贷-产品需求\{用户故事编号}-{用户故事名称}`），由步骤①确认并缓存为 `{user_story_dir}`。如会话中已缓存用户故事目录路径，直接使用；如未缓存，需先向测试人员确认用户故事信息（编号+名称）并定位/创建工作目录。

| 输出项 | 格式 | 命名 | 输出路径 | 说明 |
|--------|------|------|---------|------|
| 测试用例 Excel | .xlsx | `{需求名}_测试用例.xlsx` | `{user_story_dir}/` | ⭐ 必选 |
| 测试用例 JSON | .json | `{需求名}_测试用例.json` | `{user_story_dir}/` | ⭐ 必选 |
| 评审后快照 JSON | .json | `{需求名}_测试点_reviewed.json` | `{user_story_dir}/` | 模式A（完整流程）时生成，用于步骤⑦差异对比 |

> 💡 **模式A（完整流程）**：需要生成 `*_测试点_reviewed.json`，步骤⑦入库时对比 `*_测试点.json`（AI 原始）vs `*_测试点_reviewed.json`（评审后），计算 AI 遗漏率/通过率
> 💡 **模式B（单步模式）**：无需差异对比，只生成用例 JSON + Excel 即可

| `testcases[].original_path` | 来源路径 | ✅ | XMind 中的层级路径 |

## 脚本调用规范 ⭐

> ⚠️ **执行脚本前必须先查看参数定义，避免参数格式错误！**

### 参数查看方式

```bash
python scripts/<脚本名>.py -h
```

### 各脚本参数定义

| 脚本 | 位置参数 | 选项参数 | 正确命令示例 |
|------|---------|---------|-------------|
| parse_xmind.py | `input` (XMind文件) | `-o OUTPUT` | `python parse_xmind.py "xxx.xmind" -o "xxx.json"` |
| refine_testcases.py | `input` (测试点JSON) | `-o OUTPUT`, `--caseGroup`, `--version`, `--manager`, `--relateReqCode`, `--team`, `--product`, `--modulePath` | `python refine_testcases.py "xxx.json" --caseGroup "xxx" --version "xxx" --manager "xxx"` |
| generate_excel.py | `input` (测试用例JSON) | `-o OUTPUT`, `--caseGroup`(必选), `--version`(必选), `--manager`(必选), `--relateReqCode`, `--team`, `--product`, `--modulePath`, `--project-root` | `python generate_excel.py "xxx.json" --caseGroup "xxx" --version "xxx" --manager "xxx"` |

**⚠️ 注意事项**：
- `input` 是**位置参数**，不需要 `--input` 标志
- `-o` 是短选项，也可以用 `--output`
- `--caseGroup/--version/--manager` 在 generate_excel.py 中是**必选参数**

## 执行步骤

### 步骤 1：解析评审后的 XMind 并调用脚本转化为 JSON ⭐

> ⚠️ **这是步骤⑥的唯一数据来源！必须调用脚本转化！**

**执行逻辑**：

```
1. 搜索 XMind 文件（{需求名}_测试点.xmind，用户已在此文件上添加评审标记）
2. 调用脚本解析 XMind 并转化为 JSON：
   python scripts/parse_xmind.py "{需求名}_测试点.xmind" -o "{需求名}_测试点_reviewed.json"
   ⚠️ 注意：input 是位置参数，不需要 --input 标志！
3. 脚本解析内容：
   - 提取测试点层级结构（modules → sub_modules → testpoints）
   - 提取评审标记（marker_type：确认/新增/修改/删除）
   - 提取来源标识（source：requirement/review/tech/knowledge_base）
4. 统计评审标记分布
5. 过滤删除标记的测试点（marker_type=删除 → 不生成用例）
```

**呈现解析状态**：

```markdown
## 📄 评审后 XMind 解析状态

| 项目 | 数量 | 说明 |
|------|------|------|
| 总测试点 | N | 解析后的总数 |
| ✅ 确认 | N | 用户确认的测试点 → 生成用例 |
| 🔵 新增 | N | 用户新增的测试点 → 生成用例 |
| 🟠 修改 | N | 用户修改的测试点 → 生成用例 |
| ⚫ 删除 | N | 用户删除的测试点 → **不生成用例** |

> ⚠️ 删除标记的测试点不生成测试用例！
```

**输出文件**：`{需求名}_测试点_reviewed.json`

## 测试用例分类 ⭐

| 分类 | caseType | 侧重点 | 占比建议 |
|------|----------|--------|----------|
| 功能测试 | 功能测试 | 单模块业务、界面交互 | 60%~70% |
| 集成测试 | 集成测试 | 跨模块联调、端到端流程 | 20%~30% |
| 安全测试 | 安全漏洞 | 数据安全、权限控制 | 5%~10% |
| 性能测试 | 性能测试 | 响应时间、并发处理 | 按需 |

**判定规则：**

| 场景 | 归入分类 |
|------|----------|
| 单模块字段校验、按钮操作、页面展示 | 功能测试 |
| 涉及两个及以上模块/接口的流程 | 集成测试 |
| 涉及数据加密、权限校验、敏感数据 | 安全测试 |
| 涉及响应时间、并发、大数据量 | 性能测试 |

## 用例细化规则 ⭐

### 步骤编号规范

**必须使用「1、」「2、」格式，禁止使用「1. 」格式！**

```
✅ 正确：
1、进入页面
2、查看字段
3、验证展示

❌ 错误：
1. 进入页面    ← 禁止使用点+空格
2. 查看字段
```

### 步骤数量要求

**每个用例步骤数量 >= 2 个！**

如原有步骤只有 1 个，需补充前置操作步骤。

### 步骤与预期结果对应 ⭐

**步骤数量 = 预期结果数量，必须一一对应！**

```
✅ 正确：
步骤：
  1、点击「XXXX功能」按钮
  2、进入协议阅读页

预期结果：
  1、按钮点击成功，发起XX请求
  2、成功进入协议阅读页

❌ 错误：步骤=2，预期结果=1，不匹配
```

### 数据实例化 ⭐

**必须使用具体测试数据，禁止抽象占位符！**

| 反例 | 正例 |
|------|------|

## 注意事项

1. **输入是用户评审后的 XMind**：直接在 AI 生成的原始 XMind 上添加颜色标记，无需重命名
2. **评审标记必须解析**：正确处理新增/修改/删除标记
3. **数据必须实例化**：步骤、前置条件、预期结果使用具体值
4. **步骤编号规范**：使用「1、」格式，换行分隔
5. **步骤数量 >= 2**：每个用例至少 2 个步骤
6. **步骤与预期结果对应**：数量相等，一一对应
7. **分类占比合理**：集成测试占比 20%~40%
8. **P0 数量限制**：一个需求主流程不超过 3 个 P0
9. **前置条件完整**：包含登录状态、数据状态、配置信息
10. **预期结果可验证**：描述具体状态、值、提示信息，避免模糊表述
