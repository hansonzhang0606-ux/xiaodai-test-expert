# 测试点生成 Prompt

> 此 Prompt 用于 AI 从需求文档、评审报告、技术方案推导测试点

---

## ⛔ 禁止 AI 擅自做决定（强制按流程规范执行）

> **⚠️ 执行此 Prompt 前，必须完整阅读本文档！禁止凭记忆或推断执行！**

| 禁止项 | 说明 | 正确做法 |
|--------|------|---------|
| ⛔ 禁止假设已读取 | 不假设前置步骤已完成，必须实际执行每步读取操作 | 每个步骤按规范重新读取，不假设前置状态 |
| ⛔ 禁止简化 Todo | Todo 清单必须完整，不可省略强制检查项（⭐标记项） | 按规范创建完整 Todo 清单（10项），逐项标记状态 |
| ⛔ 禁止跳过步骤 | 强制项（⭐）不可跳过，必须逐项执行并标记状态 | 强制项必须执行，AI 无权判断跳过 |
| ⛔ 禁止自行判断 | AI 不可自行判断「不重要」「已完成」「无需执行」 | 按流程规范执行，不做任何擅自决策 |
| ⛔ 禁止假设状态 | 标记「已完成」前必须实际执行该操作，不可凭记忆标记 | 每项必须实际执行后才能标记 completed |
| ⛔ 禁止跳过整理版图片/附件信息读取 | 整理版 MD 中的图片解析和附件章节不可跳过 | 必须从整理版 MD 提取图片解析状态和附件读取状态 |
| ⛔ 禁止跳过去重合并 | 知识库补充测试点去重合并不可跳过 | 必须执行去重并记录到 dedup_results |

**典型违规场景**：

| 违规行为 | 正确做法 |
|---------|---------|
| 假设"步骤②已读过评审报告，无需再读" | 步骤④必须重新读取评审报告MD，不假设前置状态 |
| 简化 Todo 为 5 项而非 10 项 | 按规范创建完整 Todo 清单，逐项标记状态 |
| 标记"已完成"但实际未执行读取操作 | 每项必须实际执行后才能标记 completed |
| 跳过整理版 MD 中的图片/附件章节认为不重要 | 强制项必须执行，AI 无权判断跳过 |
| 知识库补充测试点不去重直接添加 | 必须执行去重合并，记录到 dedup_results |

## ⚠️ 执行前必须创建 Todo 清单 ⭐

> **AI 执行此流程时，必须先调用 `todo_write` 创建以下 todo 清单，逐项完成并标记状态！**

**步骤④子 Todo 清单（10项强制步骤）**：

| 序号 | 步骤 | 强制 | 输出/验证 |
|------|------|------|----------|
| 1 | 读取 JSON 格式规范（本文档「JSON 格式规范」章节） | ⭐ 强制 | 理解 metadata.title、modules[].testpoints 等关键字段 |
| 2 | 读取精华库 {kb_path}/patterns/精华库/common_omissions.md + improvement_patterns.md | ⭐ 默认执行 | 轻量路径：AI 错题本（≤300 Token） |
| 3 | 读取测试模式 {kb_path}/patterns/*.md + 历史文档 | 可选 | **完整路径**：用户说"结合历史"等关键词时执行 |
| 4 | 读取评审报告 MD（搜索 *_评审报告.md） | ⭐ 强制 | 不能只读 JSON，必须读 MD 文件 |
| 5 | 解析处理决策（确认处理/不处理） | ⭐ 强制 | 记录到 review_decisions |
| 6 | 推导测试点（按需求模块组织） | ⭐ 强制 | 参考整理版 MD + 评审报告 |
| 7 | 知识库补充测试点去重合并 | ⭐ 强制 | 记录到 dedup_results |
| 8 | 按格式规范生成 JSON | ⭐ 强制 | 逐字段对照验证，特别是 modules[].testpoints 格式 |
| 9 | 调用脚本生成 XMind | ⭐ 强制 | 验证脚本输出成功 |
| 10 | 生成测试点生成报告 | ⭐ 强制 | 包含精华库应用展示章节 |

> 💡 图片和附件信息从整理版 MD 中读取，不重复处理原始 source 目录。

**AI 执行流程**：
1. 收到步骤④指令 → 立即调用 `todo_write` 创建 10 项子清单
2. 逐项执行 → 每完成一项立即标记 `completed`
3. 强制项跳过 → 用户会看到 todo 状态未完成 → 立即发现问题

## 任务目标

整合分析**整理版 MD、需求评审报告**，从多维度提取测试点，生成 XMind 测试点文件用于评审。

> 💡 **整理版 MD 已包含所有源文件内容**（需求文档、技术方案、图片解析、附件），无需重复扫描。

#### 1.2 识别处理决策

**用户处理决策的常见位置**：

| 位置 | 示例 | 说明 |

### 步骤 3：去重合并 ⭐

> ⚠️ **强制约束：知识库补充测试点必须与已有测试点去重，并记录去重结果！**

---

#### 3.1 去重检查流程

**按以下顺序执行去重检查**：

```
1. 提取已有测试点
   - 从需求文档测试点提取所有测试场景描述
   - 从技术方案测试点提取所有测试场景描述
   - 从评审问题验证测试点提取所有验证场景描述

2. 遍历知识库补充测试点
   - 逐个检查是否与已有测试点语义重复
   - 逐个检查是否已被已有测试点场景覆盖

3. 判断处理
   - 语义重复 → 删除
   - 场景覆盖 → 删除
   - 新边界场景 → 保留
   - 新异常场景 → 保留
```

---

#### 3.2 去重判断标准

| 检查维度 | 判定标准 | 处理 | 示例 |
|---------|---------|------|------|
| **语义重复** | 测试目的相同 | ❌ 删除 | "征信超时处理" vs "凯旋征信超时重试" |
| **场景覆盖** | 已有测试点覆盖 | ❌ 删除 | "过期边界验证" 已覆盖 "状态有效期边界" |
| **边界补充** | 新的边界场景 | ✅ 保留 | 需求未提及的边界值 |
| **异常补充** | 新的异常场景 | ✅ 保留 | 需求未提及的异常类型 |

### 步骤 6：生成输出

> ⚠️ **必须调用脚本生成 XMind，禁止 AI 自行构造！XMind 是 ZIP 格式文件，脚本保证确定性输出！**

**执行逻辑**：

```
1. 确认测试点 JSON 已生成
2. 调用 generate_xmind.py 脚本：
   python scripts/generate_xmind.py --input "{需求名}_测试点.json" --output "{需求名}_测试点.xmind"
3. 验证 XMind 文件生成成功
4. 生成测试点生成报告.md
```

**输出文件：**

| 文件 | 格式 | 生成方式 | 命名 |
|------|------|---------|------|
| 测试点 JSON | .json | AI 生成 | `{需求名}_测试点.json` |
| 测试点 XMind | .xmind | 脚本生成（generate_xmind.py） | `{需求名}_测试点.xmind` |
| 测试点生成报告 | .md | AI 生成 | `{需求名}_测试点生成报告.md` |

## 注意事项

1. **必须搜索关联文档**：评审报告和技术方案不可跳过
2. **评审决策必须识别**：标注为"忽略"的问题不生成测试点
3. **知识库去重**：补充测试点必须与已有测试点去重

5. **图片分析**：流程图/状态图必须解析推导测试点
6. **自我评审**：生成后必须检查遗漏
