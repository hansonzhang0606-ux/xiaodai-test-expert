# 文档整理与核对 Prompt

> 此 Prompt 用于 AI 将需求目录下的源文件（Word/PDF/图片/附件）归档、转换、合并为统一 MD，供用户逐项核对。
>
> **核心理念**：AI 整理的目的是让用户核对，核对是核心，不可跳过。

---

## 两种整理入口（并行）

步骤①「文档整理」支持两种并行的内容获取方式：

| 入口 | 适用场景 | 必读文档 | 输出 |
|------|---------|----------|------|
| **A. 本地目录整理** | 需求文档以 Word/PDF/图片/Excel 等形式存放在本地目录 | 本文档 | `1.{目录名}_整理版_v{version}.md` |
| **B. Confluence 页面提取** | 需求文档在团队 Confluence 在线页面中 | `prompts/confluence_extract.md` | `1.{页面标题}_整理版_v1.0.md` |

两种入口产出的整理版 MD **完全等价**，都可以直接进入步骤②「需求评审」。

- 用户提供**本地目录路径** → 执行本文档的 12 步完整流程
- 用户提供**Confluence URL + 提取指令**（如"帮我提取这个文档内容"） → 执行 `confluence_extract.md`

> **不要混淆两个入口**：本地目录整理不做网络请求；Confluence 提取不扫描本地文件。

**步骤① 双入口流程**：

```
步骤① 双入口：
├─ 本地目录整理：document_consolidate.md
│   输入：D:\...\需求目录\
│   输出：1.{目录名}_整理版_v{version}.md
│
└─ Confluence 提取：confluence_extract.md（新增）
    输入：https://.../viewpage.action?pageId=xxx 帮我提取这个文档内容
    输出：1.{页面标题}_整理版_v1.0.md
            │
            ▼
    用户说「评审这个需求」
            │
            ▼
    步骤② 需求评审：requirement_review.md（已兼容两种来源）
```

## ⛔ 禁止 AI 擅自做决定（强制按流程规范执行）

> **⚠️ 执行此 Prompt 前，必须完整阅读本文档！禁止凭记忆或推断执行！**

| 禁止项 | 说明 | 正确做法 |
|--------|------|---------|
| ⛔ 禁止假设已读取 | 不假设前置步骤已完成，必须实际执行每步读取操作 | 每个步骤按规范重新读取，不假设前置状态 |
| ⛔ 禁止简化 Todo | Todo 清单必须完整，不可省略强制检查项（⭐标记项） | 按规范创建完整 Todo 清单，逐项标记状态 |
| ⛔ 禁止跳过步骤 | 强制项（⭐）不可跳过，必须逐项执行并标记状态 | 强制项必须执行，AI 无权判断跳过 |
|  禁止自行判断 | AI 不可自行判断「不重要」「已完成」「无需执行」 | 按流程规范执行，不做任何擅自决策 |
| ⛔ 禁止假设状态 | 标记「已完成」前必须实际执行该操作，不可凭记忆标记 | 每项必须实际执行后才能标记 completed |
|  禁止假装理解图片 | 图片解析必须分"确定/不确定"两部分 | 不能假装都理解正确，必须显式标注不确定项 |
| ⛔ 禁止自动触发评审 | 整理完成后不自动执行需求评审 | 等待用户指令决定是否评审 |

## ⚠️ 执行前必须创建 Todo 清单 ⭐

> **AI 执行此流程时，必须先调用 `todo_write` 创建以下 todo 清单，逐项完成并标记状态！**

**步骤①子 Todo 清单（13项强制步骤）**：

| 序号 | 步骤 | 强制 | 输出/验证 |
|------|------|------|----------|
| 0 | 确认用户故事信息并定位工作目录 | ⭐ 强制 | 向测试人员确认DMP用户故事编号和名称，创建/定位 `D:\效贷-产品需求\{用户故事编号}-{用户故事名称}` 目录 |
| 1 | 扫描目录，识别源文件 | ⭐ 强制 | 列出所有候选文件，按类型和时间分组 |
| 2 | 读取/创建整理索引 `source/.整理索引.yaml` | ⭐ 强制 | 增量识别：对比已归档和新增文件 |
| 3 | 输出文件清单，等待用户确认 | ⭐ 强制 | 用户确认哪些文件需要处理 |
| 4 | 原始文件复制到 `source/` 目录 | ⭐ 强制 | 默认复制归档，保留原位置文件；仅用户明确要求时才移动 |
| 5 | 更新整理索引 `source/.整理索引.yaml` |  强制 | 记录归档状态 |
| 6 | 转换文档类文件（doc/docx/pdf/ppt/pptx） | ⭐ 强制 | 调用 `convert_to_md.py` |
| 7 | 解析图片类文件（png/jpg/gif） | ⭐ 强制 | 提取结构化信息（流程/元素/状态） |
| 8 | 处理附件类文件（xlsx/json/xml） |  强制 | 转换为表格或文本 |
| 9 | 合并为 `1.{目录名}_整理版_v{version}.md` | ⭐ 强制 | 固定结构：索引→文档→图片→附件→确认清单。**文件名使用 "1.{目录名}_整理版_v{version}.md"，步骤前缀 "1." 用于标识本文件由步骤①文档整理生成** |
| 10 | 锚点溯源格式校验 | ⭐ 强制 | 确保每个章节有锚点、源文件标注、转换状态 |
| 11 | 输出核对提示，等待用户校验 | ⭐ 强制 | 不自动触发评审 |
| 12 | 根据用户反馈修正 MD | ⭐ 强制 | 用户指出错误后修正，用户确认后标记"已确认" |

**AI 执行流程**：
1. 收到步骤①指令 → 立即调用 `todo_write` 创建 12 项子清单
2. 逐项执行 → 每完成一项立即标记 `completed`
3. 强制项跳过 → 用户会看到 todo 状态未完成 → 立即发现问题

## 任务目标

将需求目录下的源文件（Word/PDF/图片/附件）归档、转换、合并为一份统一的 `{目录名}_整理版.md`，**供用户逐项核对 AI 的理解是否与源文件一致**。

**这不是评审，不是生成测试点，是纯粹的"理解核对"。**

## 步骤详情

### 步骤 0：确认用户故事信息并定位工作目录 ⭐

**执行逻辑**：

```
1. 向测试人员询问当前操作对应的 DMP 用户故事编号和用户故事名称
   - 格式：用户故事编号 + 用户故事名称
   - 示例：US-001 + 贷款审批流程优化
2. 构建工作目录路径：D:\效贷-产品需求\{用户故事编号}-{用户故事名称}
   - 示例：D:\效贷-产品需求\US-001-贷款审批流程优化
3. 检查该目录是否存在：
   - 不存在 → 自动创建该目录
   - 已存在 → 直接使用（说明"已有该用户故事目录，后续文件将存入该目录")
4. 将用户故事目录路径缓存到会话上下文（变量名：{user_story_dir}）
5. 后续所有步骤的输出文件统一存放到该用户故事目录下
```

**输出格式**：

```
📋 用户故事信息确认：

- 用户故事编号：US-001
- 用户故事名称：贷款审批流程优化
- 工作目录：D:\效贷-产品需求\US-001-贷款审批流程优化
- 目录状态：✅ 已创建 / ✅ 已存在

后续所有步骤的输出文件将存放到该目录。
```

> ⚠️ **如会话中已缓存用户故事目录路径**（测试人员之前已确认），直接使用缓存路径，跳过本步骤的询问环节，但仍需执行目录存在性检查。

### 步骤 2：读取/创建整理索引 ⭐

**整理索引用于增量识别**，记录每个文件的处理状态，避免每次全量重做。

**`source/.整理索引.yaml` 格式**：

```yaml
# 整理索引 — 用于增量识别和状态追踪
version: "v1.0"
需求名称: "额度测算V3.0"
整理日期: "2026-06-04"
MD版本: "v1.0"
MD文件: "1.{目录名}_整理版_v1.0.md"

源文件清单:
  - 文件名: "额度测算需求文档V3.0.docx"
    类型: "需求文档"
    修改时间: "2026-06-01"
    归档路径: "source/额度测算需求文档V3.0.docx"
    转换状态: "已转换"
    转换输出: "章节一"
    校验状态: "已确认"
  - 文件名: "业务流程图.png"
    类型: "流程图"
    修改时间: "2026-06-01"
    归档路径: "source/images/业务流程图.png"
    转换状态: "AI解析"
    转换输出: "章节四-4.1"
    校验状态: "已确认"
    AI确定度: "4/5节点确定"
  - 文件名: "数据字典.xlsx"
    类型: "数据附件"
    修改时间: "2026-06-01"
    归档路径: "source/attachments/数据字典.xlsx"
    转换状态: "已转换"
    转换输出: "章节五-5.1"
    校验状态: "已确认"
```

**增量识别逻辑**：

```
1. 读取 source/.整理索引.yaml（如存在）
2. 对比"源文件清单"和"当前目录实际文件"
3. 识别三种状态：
   - ✅ 已归档：在索引中，且在 source/ 中存在
   - 🆕 新增：在目录中，但不在索引中
   - ⚠️ 已删除：在索引中，但目录中已不存在（提醒用户）
4. 只处理新增文件，已归档文件跳过
5. 更新索引，版本号递增（v1.0 → v1.1）
```

### 步骤 5-8：转换处理 ⭐

#### 5. 文档类转换（doc/docx/pdf/ppt/pptx）

```bash
python {技能目录}/scripts/convert_to_md.py <文件路径>
```

- 转换结果标记为"机械转换"，AI 不做内容修改
- 转换失败时记录失败原因

#### 6. 图片类解析（png/jpg/gif）

**️ 图片解析原则：能还原的尽量还原，不确定的显式标注**

> **核心原则**：AI 应尽力将流程图、时序图、状态机图还原为**结构化文字流程图（步骤+缩进）**，而非扁平表格或复杂图形。不确定的分支/连线/触发条件用 `??` 标记标注，用户重点核对。

**图片解析策略**：

| 图片类型 | AI 优先输出 | AI 做不到什么 | 降级方案 |
|---------|-----------|-------------|---------|
| **业务流程图** | 步骤+缩进文字流程图 | 无法100%确认连线关系、分支条件 | 不确定分支用 `??` 标注 + 表格补充不确定项 |
| **时序图** | 步骤+缩进文字时序 | 无法100%确认调用顺序、同步/异步 | 不确定调用用 `??` 标注 + 表格补充 |
| **状态机图** | 步骤+缩进文字状态流转 | 无法100%确认流转触发条件 | 不确定流转用 `??` 标注 + 表格补充 |
| **界面原型图** | 页面元素表格 | 无法理解交互逻辑、页面跳转 | 保持表格（无流程可还原） |
| **数据模型图** | ER 关系表格 | 无法100%确认外键关系 | 不确定关系用 `??` 标注 |

**步骤+缩进流程图规范（首选方案）**：

> ⭐ **优先使用此方案**：纯文本结构，任何 Markdown 查看器均可正常显示，无需浏览器渲染 Mermaid。
